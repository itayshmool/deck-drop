Cross-Validation Pipeline

Two independent LLMs, one deterministic verdict

LLM #1 — Wolf Agent

Role: Primary analyst. Finds structural issues.

Output schema:

finding severity recommendation impactEstimate

Sees: anonymized summary only

LLM #2 — Mirror Agent

Role: Adversarial challenger. Stress-tests every finding.

Output schema:

agrees challenges[] riskFlags[] alternativeView

Sees: same summary + Wolf's findings

Agents run independently — no shared context, no chain-of-thought leak ↓

Consensus engine is pure JS — 7 deterministic rules, zero LLM in the verdict loop

Consensus Engine

No LLM decides the final verdict

#ConditionVerdict
1Mirror agrees + confidence ≥ 80%VALIDATED
2Mirror agrees + confidence 60–79%VALIDATED (reduced)
3Mirror partially agrees, flags risksCONTESTED
4Mirror disagrees but Wolf confidence ≥ 90%CONTESTED
5Mirror disagrees + Wolf confidence < 60%NEEDS_DATA
6Mirror provides counter-evidenceNEEDS_DATA
7Mirror disproves or confidence < 40%REJECTED

Output per insight: consensus tag + merged confidence + severity + synthesized summary + recommendations

Zero-Trust Data Flow

What data goes where

📄

Raw Data (XLSX/CSV) — uploaded by user, stored temporarily

↓
✂

PII Stripper — removes name, email, SSN fields from raw_data JSON

↓
🔐

PII Encryptor — AES-256-GCM on employees.name column (unique IV per record)

↓
📊

Pre-Processor — outputs anonymized summary: counts, ratios, distributions. No names. No individuals.

↓
🤖

LLM Agents — receive ONLY the anonymized summary. Never see raw data, PII, or individual employees.

PII Encryption

Defense in depth for sensitive data

Layer 1 — Strip

stripPII() removes name/email/SSN from raw_data JSON before DB write

Regex-based field detection — runs on every record

Layer 2 — Encrypt

AES-256-GCM on employees.name column

256-bit key · unique 12-byte IV per record · auth tag stored alongside

Layer 3 — Anonymize

Pre-processor outputs only aggregate statistics

Dept headcounts, not rosters · Distributions, not org charts

// Encryption format stored in DB
"iv:authTag:ciphertext"  // base64 encoded

// Example
"mK9x2f...:j8Hp3...:aB7kLm..."  →  "Lionel Messi"

Multi-Tenant Isolation

Every query scoped to tenant

-- Every table has tenant_id column
SELECT * FROM employees
WHERE tenant_id = $1 AND dataset_id = $2;

-- PostgreSQL RLS policies (safety net)
CREATE POLICY tenant_isolation ON employees
  USING (tenant_id = current_setting('app.tenant_id')::uuid);

Queue Architecture

BullMQ + Redis — reliable job processing

analysis-pipeline

Concurrency: 2 parallel workers

Rate limit: 5 jobs / minute

Retries: 3 attempts, exponential backoff

Runs: Wolf + Mirror + Consensus

fix-plan-pipeline

Concurrency: 1 worker

Rate limit: none (on-demand)

Retries: 2 attempts, 5s base backoff

Runs: Fixer agent

restructure-pipeline

Concurrency: 1 worker

Rate limit: none (on-demand)

Retries: 2 attempts, 5s base backoff

Runs: Architect agent

// Job lifecycle
API → db.insert(status: "queued")
    → queue.add({ analysisId, tenantId })
    → Worker picks up → status: "running"
    → Success → status: "completed" + store results
    → Failure → retry 3x → status: "failed"

API Design

Hono middleware stack

→ Request In
→ securityHeaders HSTS, CSP, X-Frame-Options
→ cors Explicit origins, preflight cache
→ requestId X-Request-Id on every response
→ errorHandler Catch-all, structured JSON errors
→ tenantMiddleware Resolve X-Tenant-Id header
→ authMiddleware Bearer token validation
→ rateLimiter 100 req/min API, 5/5min analysis
→ auditMiddleware Log action to audit_log
→ Route Handler

Route ordering matters in Hono: specific routes (e.g. /org-tree) must come before catch-all /:param routes.

Typed context: AppEnv interface ensures c.get("tenantId") is always available after middleware.

Zod validation on all mutation endpoints — reject bad input before it reaches business logic.

File upload: 50MB limit, xlsx/csv/pdf, validated at middleware level.
